路由设计

发表于 2018-03-22 | 分类于 Weex

初衷

产生对原生应用进行路由设计的初衷，是在很久以前移动端应用还在hybrid时。我们引用了一个移动端跨平台开发的框架weex，用于取代原来html5维护的业务。在原来html5的方案中，我们通过scheme来唤起原生页面。



后来发现在处理以下这些业务场景时，以上方案存在了种种掣肘

需求
  1. 接收通知，打开应用内的某个页面
  2. html5维护的页面跳转到原生页面
  3. 页面降级，在低版本手机降级到h5
  4. 屏蔽掉一些不合法的链接
弊端
  • 显示：startActivity需要知道目标界面class，导致类耦合太大
  • 隐式：需要向目标界面传递参数，传参的key变动。调用方及使用方都需变动
  • 符合Android.manifest的scheme链接都可以直接打开
  • 无法动态拦截跳转，动态降级
  • Android, H5, Ios三端协议皆不一致

并且在引入了weex方案之后，因为社区和类库的不完善。没有一个路由解决方案。这使得我们在非原生和原生之间跳转出现了问题，同时原有scheme方案灵活性不够的弊端也逐渐凸显。
另一个方面的原因是由于应用页面的增多，项目逐渐开始变得臃肿不堪。我们开始着手对代码进行模块化隔离，而各模块之间由于解除了依赖关系，所以模块间跳转也出现了问题。

双向绑定

发表于 2017-12-05 | 分类于 Vue2.0

前言

在Vue1.0到Vue2.0的版本移除了双向绑定，这会使得组件状态的变动不可预测。因此我们不再能够在child之中直接操作parent传递下来的属性。这么做的好处是单向数据流使得组件的状态变动可预测，但不可避免的是子组件需要通过事件回传更改后双向绑定属性的值。

2.3版本中双向绑定的写法

本质上是vue-loader在编译prop.sync的语法糖时生成了下面代码
<child :show="value" @on-show-change="(newVal)=>value=newVal"></child>

以上仅是举了一个简单的例子，对于复杂状态的组件，可能存在多个位置对属性进行更改。一般会将需要操作的属性丢在data中clone一份，组件对备份后的属性进行使用及操作

需要注意一点的是，data中声明的字段不再能够以_或$开头。所以这里以p_定义备份的属性字段,子组件内部通过操作备份字段触发绑定属性的变动
